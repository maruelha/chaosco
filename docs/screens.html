Core
Dashboard GET /

The home screen. Shows a grid of action cards — one per major module. Acts as the main navigation hub since there is no persistent sidebar.

Cards Import, Defects, Core South Spillover, Retail, Meeting Prep, To-Do, Follow-ups, Links, CS Follow-Up Tracker Primary action Run Import — triggers a POST to /import and redirects to the Import Result screen Navigation Each card contains a link button to the corresponding list screen. The site title in the header also links back here from every page.
Import Result POST /import

Shown after running an import. Displays a per-tab summary of what happened — how many rows were inserted, updated, or skipped, plus archive status and any skip-log file path.

Per-tab summary Defects · Core South Spillover · Retail (only tabs with enabled: true in settings.yaml appear) Counts shown Inserted · Updated · Skipped (blank key) · Skipped (duplicate) · Ignored (blank rows) Archive info Archive status (archived / skipped_duplicate / disabled) and path of archived file Skip log Path to the per-run CSV if any rows were skipped Navigation Back to Dashboard
Defects
Defects — List GET /defects

Paginated list of all imported defects with inline filters. The default view hides statuses "Withdrawn" and "Confirmed" (configurable in settings.yaml).

Columns Excel Row · Defect ID · Channel · Country · SolMan Name · Status · Priority · Assigned To · Prod · Blocked TCs · Notes · Action Blocked TCs column Count of Retail test cases whose defect_id_ref references this defect. Computed as a subquery (LIKE match — not stored). Shows — when zero. When non-zero, displayed as a red badge link that opens the Retail list pre-filtered by that defect ID (?search_defect=<id>). Filters Search (defect ID) Channel Status (multi-select) Action needed Show all toggle Row actions Click row → Defect Detail · Add note shortcut per row · Blocked TCs badge → Retail list filtered by defect ID Flash banners Green "Note added." banner when returning from a successful note add Table source defects LEFT JOIN defect_annotations + subquery on retail.defect_id_ref
Defects — Detail GET /defects/<defect_id>

Full view of a single defect — imported fields on the left, editable annotation form on the right, and a notes log at the bottom.

Read-only fields All columns from the defects table (channel, status, priority, dates, scenario, etc.) Editable annotation Description · Business Impact · Reach · Retest Needs · Next Step · Action Needed (checkbox) · Comments
Save → POST to same URL → redirect with ?saved=1 flash Notes section List of all notes for this defect (heading, body, timestamp). Per-note Edit and Delete links. Each note also shows image thumbnail gallery with a 📷 Add screenshot button. Hover a thumbnail to reveal the ✕ remove button. Ctrl+V pastes a clipboard image (e.g. from Snipping Tool) into the last-hovered note. Flash banners Annotation saved · Note added · Note saved · Note deleted Navigation ← Back to Defects · Add note Table source defects LEFT JOIN defect_annotations + notes + attachments
Defects — Note Add / Edit / Delete GET /defects/<id>/notes/add  ·  /notes/<id>/edit  ·  /notes/<id>/delete

Three separate screens for managing notes on a defect. Add and Edit share the same note_form.html template (distinguished by a mode variable). Delete is a confirmation page. Screenshot attachments are managed via AJAX on the detail page itself (no separate screen).

Add / Edit fields Heading (optional) · Note text (required — shows inline error if empty) Delete Confirmation page showing the note heading/body. Requires a second POST to confirm deletion. Return flow Add can return to either the defect detail or the defects list (controlled by return_to query param). Edit and Delete always return to the defect detail. Screenshot attachments Upload via file picker or Ctrl+V paste from Snipping Tool (AJAX, no page reload). Delete via ✕ button on thumbnail hover. Files stored in data/uploads/. Routes: POST /notes/<note_id>/attachments/add · POST /notes/<note_id>/attachments/<id>/delete · GET /uploads/<filename>. Tables written notes (entity_type = 'defect') · attachments (on upload/delete)
Core South Spillover
Spillover — List GET /spillover

Frozen-pane scrollable table of all Core South Spillover items. All annotations are edited inline without leaving this page — saves happen via AJAX.

Columns Excel Row · Type · Name · Country · Area · Status · Assigned To · External ID · Importance for Sign-Off · Critical for Sign-Off · Sign-Off Group · Next Step · Comment for Sign-Off Filters Area (multi) Type (multi) Status (multi) Assigned To (multi) Critical for sign-off (multi) Show passed & dropped toggle Hidden by default Statuses: Passed · Passed pending Solman · Dropped (configurable in settings.yaml) Inline edits (AJAX) Importance for sign-off Next step Comment for sign-off Sign-off group Critical for sign-off (each field saves on blur/change to POST /spillover/<id>/annotation or /critical) Comment popup Per-row "Comment History" button opens a panel showing timestamped authored comments and a text area to add a new one. Saves via POST /spillover/<id>/comment. Table source spillover LEFT JOIN spillover_annotations
Retail
Retail — List GET /retail

Filterable table of all Retail test-case rows. Next steps are editable inline; comment history opens in a popup. No status-hiding — all rows visible by default.

Columns Excel Row · Test Case ID · Country · Scenario · Testcase Name · Status · Assigned To · Key User · Next Step · Action Needed · Note count Dropdown filters Status (multi) Assigned To (multi) Country (multi) Scenario (multi) Action needed Text searches Defect ID ref Order number S4 Billing document Inline edit (AJAX) Next step — editable on row; saves on blur to POST /retail/<id>/annotation Comment popup Per-row "Comment History" button; saves via POST /retail/<id>/comment Row link Click row (or test case ID) → Retail Detail screen Header links → Report link to the Retail Status Report · → Sign-Off Report link to the Retail Spillover Report Table source retail LEFT JOIN retail_annotations
Retail — Detail GET /retail/<retail_id>

Full view of a single retail test case. Shows all imported fields plus an annotation section and a notes log at the bottom.

Read-only fields All columns from the retail table: status, scenario, all S4 doc numbers, order refs, defect ref, evidence links, dates, etc. Editable annotation Next Step · Comment History · Action Needed (checkbox)
Save → POST → redirect with ?saved=1 flash Notes section List of notes (heading, body, timestamp). Per-note Edit and Delete links. Each note shows image thumbnail gallery with a 📷 Add screenshot button. Hover a thumbnail to reveal the ✕ remove button. Ctrl+V pastes a clipboard image (e.g. from Snipping Tool) into the last-hovered note. Flash banners Annotation saved · Note added · Note saved · Note deleted Navigation ← Back to Retail · Add note Table source retail LEFT JOIN retail_annotations + notes + attachments
Retail — Note Add / Edit / Delete /retail/<id>/notes/add  ·  /notes/<note_id>/edit  ·  /notes/<note_id>/delete

Same structure as the Defect note screens — Add and Edit share note_form.html; Delete is a confirmation page. Always returns to Retail Detail. Screenshot attachments are managed via AJAX on the detail page (no separate screen).

Add / Edit fields Heading (optional) · Note text (required) Screenshot attachments Upload via file picker or Ctrl+V paste (AJAX). Delete via ✕ on thumbnail hover. Same routes as Defects: /notes/<note_id>/attachments/… Tables written notes (entity_type = 'retail') · attachments (on upload/delete)
Retail — Status Report GET /retail/report

Live report that maps raw retail.status values to named buckets defined in config/status_mappings.yaml. Shows total counts per bucket and overall. Provides two export options.

Buckets Back with Sales · With DTC · In Progress with DTC · Passed with DTC · Incoming (Gatekeeper) · Ready for Validation · In Progress · In Clarification · Blocked Totals Count per bucket + overall total row count Export actions Save to Excel — appends one row to output/retail_report_log.xlsx (AJAX, date auto-filled with today)
Download HTML — downloads a standalone retail_report_YYYY-MM-DD.html snapshot Config Bucket definitions in config/status_mappings.yaml; Excel log path in config/settings.yaml → retail_report_xlsx Table source retail (GROUP BY status)
Sign-Off Reports
Retail Spillover Report GET /report/retail

Printable sign-off summary for Retail spillover items (excludes ECOM and Omni areas). Groups items by their Critical for sign-off value.

Sections Critical: Yes · Critical: Slightly · Critical: No · Not set — each section sorted by sign-off group then name Item fields shown Name · Type · Country · Area · Status · External ID · Importance for Sign-Off · Next Step · Comment for Sign-Off · Sign-Off Group Hidden statuses Same as spillover list (Passed / Dropped, configurable in settings.yaml) Filter Excludes rows where area is "ecom" or "omni" Table source spillover LEFT JOIN spillover_annotations
ECOM / Omni Spillover Report GET /report/ecom

Same layout as the Retail Spillover Report but filtered to ECOM and Omni areas only. Includes an additional section for known production defects.

Sections Same critical-for-signoff grouping as Retail report Filter Only rows where area is "ecom" or "omni" Extra section Known Production Defects — all rows from known_prod_defects, shown at the bottom Table source spillover LEFT JOIN spillover_annotations + known_prod_defects

Planning
Meeting Prep GET /meeting-prep

Tracks agenda topics per named meeting. Topics are added inline via a quick-add form at the top; status changes happen in-row via AJAX. No detail page — everything is managed on this one screen.

Columns Meeting · Topic · Status · Inline note · Notes count Filters Meeting (dropdown) Status (dropdown) Meeting options Balazs · GPO · Sync&Solve · DTC O2C Daily · Sales ECOM daily · Other Status values planned · discussed · parked · dropped Quick-add form Meeting selector + topic text → POST /meeting-prep/add Inline actions (AJAX) Status change → POST /meeting-prep/<id>/status
Inline note → POST /meeting-prep/<id>/note
Notes popup → load/add via /meeting-prep/<id>/notes Table source meeting_prep LEFT JOIN notes
To-Do List GET /todos

General-purpose task list with priority, due date, kind classification, and owner. Default view hides closed items. Sorted by: blocked first, then in_progress, then open, then by priority and due date.

Columns Status · Priority · Area · Kind · Topic · Due Date · For Whom · Notes count Filters Area Status Priority For Whom Due Date Show closed toggle Status values open · in_progress · blocked · closed Priority values High · Medium · Low Quick-add form Area + Kind + Topic + Priority + Due Date + For Whom → POST /todos/add Inline actions (AJAX) Status change → POST /todos/<id>/status
Edit row → modal/form → POST /todos/<id>/edit
Notes popup → load/add via /todos/<id>/notes Due date highlight Overdue dates shown in red (compared against today's date passed from server) Table source todos LEFT JOIN notes
Follow-ups GET /followups

Lightweight "chase" list — who needs to be followed up with, on what topic, and by when. Done items hidden by default. Sorted by: in_progress first, then open, then by next date.

Columns Status · With Whom · Topic · When Next · Notes count Filters With Whom When Next (date) Show done toggle Status values open · in_progress · done Quick-add form With Whom + Topic + When Next → POST /followups/add Inline actions (AJAX) Status change → POST /followups/<id>/status
Notes popup → load/add via /followups/<id>/notes Table source followups LEFT JOIN notes

Global Widget
Enhancements Panel Available on every page (in base.html)

A floating "☆ Enhancements" button in the bottom-right corner of every page opens a slide-out panel for managing the tool enhancement backlog. No page navigation required.

Access Click "☆ Enhancements" button → panel opens; click ✕ or the button again to close Add form Area · Enhancement text · Priority → POST /enhancements/add (AJAX, no page reload) Filters (in panel) Area Priority Status Show closed toggle Per-item action Status dropdown per item → POST /enhancements/<id>/status (AJAX) Status values not_started · in_progress · closed Data source GET /enhancements → JSON (loaded each time the panel opens or filters change) Table source enhancements
