v2 Blueprint — Clean Rebuild of chaosco

Founding document · rebuild clean, test-first, reusing the existing database & schema

Why this is safe (not a risky rewrite): the valuable asset — the schema + data + importer logic — survives. Only the messy, cheap-to-reproduce part (routes, templates, copy-pasted data-access) is rebuilt. That inverts the usual “never rewrite” risk. The old app/ keeps running as your daily tool until v2 is ready.

1 · Goals — the four axes

AxisGoalHow v2 delivers it
ChangeabilityEasy to modifyLayered architecture + one generic entity pattern (no per-entity copy-paste)
VerifiabilityChange without fearTest-first; characterization tests on ported importers
DurabilityData & env surviveVersioned migrations, startup DB backup, dev-on-a-copy, pinned deps
RobustnessFails gracefullyValidation at the Excel boundary + the service layer
Restraint clause. Single-user local tool — v2 deliberately omits auth, multi-user concurrency, containers, microservices, caching, API versioning. Knowing what to leave out is part of the architecture. Clean ≠ over-abstracted; avoid the second-system effect.

2 · Strategy: same repo, isolated v2/

chaosco/
  app/      ← OLD app, untouched, keeps running (+ Inbox + small view added here)
  v2/       ← NEW app, built clean
  docs/

Two hard rules:

  1. v2/ never imports from app/ — old code is reference you copy from, never a dependency.
  2. v2/ runs against a copy of test_coordination.db during dev — a v2 bug can’t corrupt real authored data. Adopt the live DB only at cutover.
Cutover checklist (define “done” up front so you don’t stall at 80%):
  • v2 import produces byte-identical DB rows to the old app (proven by tests)
  • All daily-used screens exist in v2
  • v2 points at the live DB; old app/ archived (moved out, not deleted, for one safety period)

5 · The layers (your “orchestrator in the middle”)

  web/routes   →   services   →   db (repository / notes / schema)
  (thin: HTTP,     (business      (SQL only;
   parse,render)    logic, the     no business logic)
                    orchestrator)
  • web/routes — thin. Parse request, call a service, render. No SQL, no logic.
  • services — the middle layer you wanted. Validation, “what should happen,” orchestration. The rules live here.
  • db — SQL only. repository.py generic; notes.py unified; schema.py/migrations.py own structure.

6 · The duplication-killer: one generic entity pattern

Declare each simple entity once; generic machinery does CRUD, routes and rendering:

# models.py (illustrative)
CONTACTS = Entity(
    table="contacts",
    fields=["name","email","area","topic","comments","tags"],
    filters=["area","topic","tags"],
    search=["name","email"],
)

Adding a new simple entity = one declaration, not 6 functions + 6 routes + a 250-line template.

Where to STOP. Imported verticals (defects, spillover, retail) have real bespoke logic — keep their importers and specialized queries hand-written. Use the generic pattern for simple CRUD entities and annotations. Let special things be special; don’t force everything through one abstraction to be clever.

7 · Test-first plan

  1. Characterization tests on ported importers first. Feed a sample Excel, assert the rows. Prove v2 importers produce identical output to the old app — the safety net for the part you copy.
  2. Service tests — filing logic, CRUD rules, report buckets.
  3. Web smoke tests — each route returns 200 with the right data.

Not 100% coverage — fearless change on the critical paths.

8 · Durability (folded in, not a phase)

  • db/migrations.py — ordered & versioned; no scattered ALTERs.
  • Startup DB backup — timestamped copy on launch (cheap insurance).
  • Dev-on-a-copy — protects authored data during the messy build.
  • Pin requirements.txt — reproducible a year from now.

9 · Build order

  1. Scaffold v2/ (app factory, config, DB copy) + one passing test.
  2. Port importers + characterization tests → prove identical output.
  3. db layer — schema, connection, generic repository, port notes.
  4. services — imports, entities, notes_service, retail_report.
  5. web — base + macros + generic entity pages + dashboard; one shared attachments.js.
  6. Migrate each vertical onto the pattern, one at a time (tests stay green).
  7. Cutover per §2.

10 · First feature in v2: Inbox (clean version)

The Inbox you ship today in the old app is throwaway plumbing — but its data design carries over for free (just notes rows with entity_type='input'). In v2, Inbox becomes the first real exercise of the clean stack: notes_service.file_item() + generic note routes + the shared note-log macro. Small, self-contained, the perfect pilot. Data design lives in inbox_module_plan.md — already right, unchanged.
