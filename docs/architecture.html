chaosco — Architecture

Local Flask app · SQLite database · Excel as read-only data source · Updated 2026-06-25

Purpose in one paragraph

chaosco is Marina's personal UAT coordination toolkit for an SAP S/4 retail migration go-live. The team tracks defects in SolMan, test-case execution in a shared Excel workbook, and sign-off decisions in various lists. chaosco reads that shared Excel file (read-only), imports it into a local SQLite database, and serves a private Flask browser UI at http://127.0.0.1:5000. The UI adds annotation layers (next steps, comments, flags, notes) on top of the imported data without ever writing back to Excel. It also stores personal coordination data (to-dos, follow-ups, meeting prep, encouragements) that lives entirely in the local database.

Non-negotiable architecture rules

RuleWhat it means in practice
SQL only in database.py app/web.py never writes SQL directly. Every read and write goes through a function in database.py. This makes the DB contract visible in one place and keeps routes thin.
Excel is read-only No importer or route ever writes to the Excel file. chaosco is a personal read layer on top of the shared workbook.
Imported data and authored data never mix Importers write only to their primary tables (defects, spillover, retail). User annotations live in separate tables (defect_annotations, spillover_annotations, retail_annotations). An import run can never overwrite a note, flag, or next-step.
One notes table for everything Every module that needs a notes log — defects, retail, todos, follow-ups, shelf, inbox, etc. — writes to the single notes table using (entity_type, entity_id) as the address. Never create a module-specific notes table.
Each Excel tab = its own importer + its own DB table Defects, Spillover, and Retail each have a dedicated importer module. When ECOM/Omni are added, they get their own importer and table. Importers are never modified to touch another importer's table.
Config-driven, not code-driven File paths, sheet names, and hidden statuses live in config/settings.yaml. Retail report bucket definitions live in config/status_mappings.yaml. Changing a status label or folder path requires no code edit.
Retail is the template vertical When adding ECOM or Omni, copy the Retail importer, Retail annotation table, and Retail UI screens as the starting point. Do not invent a new pattern.

Import is idempotent — safe to run multiple times. Each run upserts rows by match key: first_seen is set once and never overwritten; last_seen updates every run. Rows are never deleted by an import. The source file is copied to an archive folder on each run.

Match keys per vertical:

  • Defects: defect_id (SolMan ID, TEXT primary key)
  • Spillover: excel_row cast to TEXT (stable after 2026-05 migration)
  • Retail: lower(test_case_id) || "||" || lower(country)

Shared patterns — plug-in modules

Notes module

One notes table for all entity types. Address = (entity_type, entity_id). Adding notes to any new module requires:

  • Call database.add_note(conn, entity_type, entity_id, heading, text)
  • Call database.list_notes(conn, entity_type, entity_id) to fetch
  • Add note routes to web.py (add / edit / delete)
  • Render the notes block in the detail template

Never create a module-specific notes table. The shared table is the only correct path.

Attachments module

Files (images + documents) link to notes.id, not to entity rows directly. This means attachments automatically follow a note when it is re-parented (e.g. when an inbox item is filed to Shelf or another entity).

  • Files stored in data/uploads/<note_id>_<timestamp>_<name>
  • Upload: POST /notes/<note_id>/attachments/add
  • Delete: POST /notes/<note_id>/attachments/<id>/delete — removes DB row + disk file
  • Serve: GET /uploads/<filename>

Order details module

Generic order-number log, mirrors the notes pattern. Any module can attach order lines without a dedicated table. Currently used by Spillover and ECOM Gatekeeper.

  • Address = (entity_type, entity_id)
  • Columns: order_type · order_number · comment · docs_in_s4 (S4 confirmed flag)
  • Routes: GET/POST /order-details/<entity_type>/<entity_id>[/add]

Inbox + filing

Unfiled items are notes rows with entity_type='input' and entity_id='inbox'. Filing is a single SQL UPDATE:

  • UPDATE notes SET entity_type=?, entity_id=? WHERE id=?
  • All attachments follow automatically (linked to note_id)
  • Filing to Shelf additionally creates a new shelf row first, then re-parents the note to it

Import vertical — how to add a new one (ECOM / Omni)

Use Retail as the template. Steps:

  1. Create app/ecom_importer.py — copy retail_importer.py, adjust column mappings and match key
  2. Add CREATE TABLE IF NOT EXISTS ecom (...) to database.init_db()
  3. Add CREATE TABLE IF NOT EXISTS ecom_annotations (...) to database.init_db()
  4. Add database.upsert_ecom() — copy upsert_retail(), adjust for new table
  5. Wire the importer into app/importer.py
  6. Add routes to app/web.py — copy Retail list + detail + annotation routes
  7. Create templates — copy retail_list.html and retail_detail.html as starting points
  8. Add dashboard card; update docs/screens.html, docs/database_schema.html, CLAUDE.md

ECOM orders already captured in order_details with entity_type='ecom_gatekeeper' can be re-pointed to the new ECOM rows via UPDATE order_details SET entity_type='ecom', entity_id=<ecom_id> — no data copy needed.

Database design principles

Imported vs authored data separation

Every imported table (defects, spillover, retail) has a paired annotations table (defect_annotations, etc.). Import runs upsert the primary table; they never touch the annotations table. A user's next step or flag is never overwritten by a re-import.

Migrations via ALTER TABLE

When a new column is added to an existing table, it is added via ALTER TABLE ... ADD COLUMN in a migration block at the bottom of init_db(). These are safe to re-run (they use IF NOT EXISTS equivalents or are wrapped in try/except). Column additions are tagged migration in the schema doc.

All datetimes as TEXT (ISO 8601)

SQLite has no native datetime type. All timestamps are stored as ISO 8601 TEXT strings (YYYY-MM-DDTHH:MM:SS). Sorting and comparison work correctly with TEXT comparison on ISO strings.

Entity-type + entity-id addressing

The shared tables (notes, order_details) use a (entity_type TEXT, entity_id TEXT) pair as the address instead of typed foreign keys. This lets any new module plug into shared infrastructure without a schema change. entity_id is always the TEXT representation of the primary key.

PDF export

app/pdf_utils.py provides two reusable helpers:

  • render_pdf(html: str, filename: str) → Response — generates an A4 PDF and returns it as a Flask download response. Used by report PDF routes.
  • save_pdf(html: str, filepath: Path) → None — writes the PDF to disk. Used by the planned report export feature.

WeasyPrint is imported lazily — the app starts and runs normally without it installed. Only the PDF routes fail if it is absent.

To add PDF export to any new report: render the existing download template to a string, pass to render_pdf() or save_pdf(). No per-report boilerplate.

How to start the app

  • Windows (normal use): double-click run_web.bat — kills any existing process on port 5000, then starts Flask
  • Command line: python -m app.web from the project root
  • URL: http://127.0.0.1:5000
  • Database: auto-created at data/test_coordination.db on first start via database.init_db()
