chaosco — Architecture

Local Flask app · SQLite database · Excel + Jira XML as read-only data sources · Updated 2026-07-18 (post-refactor: feature route modules + app/db package)

Purpose in one paragraph

chaosco is Marina's personal UAT coordination toolkit for an SAP S/4 retail migration go-live. The team tracks defects in SolMan, test-case execution in a shared Excel workbook, and sign-off decisions in various lists. chaosco reads that shared Excel file (read-only), imports it into a local SQLite database, and serves a private Flask browser UI at http://127.0.0.1:8010. The UI adds annotation layers (next steps, comments, flags, notes) on top of the imported data without ever writing back to Excel. It also stores personal coordination data (to-dos, follow-ups, meeting prep, encouragements) that lives entirely in the local database.

Non-negotiable architecture rules

RuleWhat it means in practice
SQL only in the storage layer Routes never write SQL. Every read and write goes through a function in the app/db/ package (reached via the app/database.py facade). This makes the DB contract visible in one place and keeps routes thin.
Excel is read-only No importer or route ever writes to the shared Excel file (the only local Excel WRITE is the append-only report log workbook in output/). chaosco is a personal read layer on top of the shared workbook.
Imported data and authored data never mix Importers write only to their primary tables (defects, spillover, retail, ecom, jira_issues). User annotations live in separate tables (defect_annotations, …, ecom_annotations, gatekeeper_annotations). An import run can never overwrite a note, flag, or next-step.
One notes table for everything Every module that needs a notes log writes to the single notes table using (entity_type, entity_id) as the address, served by the registry-driven generic Blueprint web_notes.py (+ shared _notes_section.html and static/notes.js). Never create a module-specific notes table, route set, or attachment script.
Each Excel tab = its own importer + its own DB table Defects, Spillover, Retail, and ECOM each have a dedicated importer module and table. Importers are never modified to touch another importer's table.
Config-driven, not code-driven File paths, sheet names, hidden statuses, expected reporters, backup folder etc. live in config/settings.yaml (machine-specific overrides in gitignored settings.local.yaml — MERGED per top-level key over the base since 2026-07-05, local wins). Report bucket definitions (shared by Retail AND ECOM) live in config/status_mappings.yaml.
UI from components Templates import _macros.html (page header, stat cards, sections, tables, pills) and use the component classes in style.css. Shared behaviors are drop-in includes (_notes_section, _order_details, _comment_history_dialog, _row_validation_dialog, _connections, _entity_links, _issue_message, _next_step_history, _teams_chat_links). Never copy HTML between templates.
The tracker is the reference implementation for new modules New modules follow the Blueprint pattern (own app/db/ or db_*.py storage + own web_*.py Blueprint registered in app/web.py + macro-built templates + a notes-registry entry + tests). The Retail Requirements Tracker shows the full pattern end to end.

Data flow — import path

Excel workbook
→
pandas + openpyxl
→
vertical importer
(read_defects / spillover_importer / retail_importer)
→
database.upsert_*()
→
SQLite

Import is idempotent — safe to run multiple times. Each run upserts rows by match key: first_seen is set once and never overwritten; last_seen updates every run. Rows are never deleted by an import. The source file is copied to an archive folder on each run.

Match keys per vertical:

  • Defects: defect_id (SolMan ID, TEXT primary key)
  • Spillover: excel_row cast to TEXT (stable after 2026-05 migration)
  • Retail: lower(test_case_id) || "||" || lower(country)
  • ECOM: lower(trim(jira_id)) — rows without a JIRA ID are skipped

Second import path — Jira XML (2026-07-11): "↻ Update from Jira" on the Gatekeeper AND ECOM pages runs ONE unified import (app/jira_importer.py): newest .xml in jira_folder → shared jira_issues/jira_comments store. Tracked tickets always refresh their living fields (status, assignee, acceptance criteria, comments, reporter); new tickets enter only when assigned to Marina or already on the ECOM board. Third path — SolMan export → solman_sync.py updates defects.solman_status/assigned_to only.

Shared patterns — plug-in modules

Notes module (registry-driven)

One notes table for all entity types. Address = (entity_type, entity_id). The generic Blueprint web_notes.py serves add/edit/delete for EVERY registered type (/n/<type>/<id>/…). Adding notes to a new module:

  • Add one entry to web_notes.REGISTRY
  • {% include '_notes_section.html' %} in the detail template

Done. Never create module-specific notes tables, routes, or JS — attachments/paste come from the shared static/notes.js.

Attachments module

Files (images + documents) link to notes.id, not to entity rows directly. This means attachments automatically follow a note when it is re-parented (e.g. when an inbox item is filed to Shelf or another entity).

  • Files stored in data/uploads/<note_id>_<timestamp>_<name>
  • Upload: POST /notes/<note_id>/attachments/add
  • Delete: POST /notes/<note_id>/attachments/<id>/delete — removes DB row + disk file
  • Serve: GET /uploads/<filename>

Order details module

Generic order-number log, mirrors the notes pattern. Used by Spillover and the Jira world — Gatekeeper tickets and the ECOM board share the SAME rows at address ('jira', jira_key) (2026-07-16). Grouped archive batches live in order_details_history (📦 archive-as-group). The ⤵ take-over pulls labeled order numbers from a ticket's Jira acceptance criteria.

  • Address = (entity_type, entity_id)
  • Columns: order_type · order_number · comment · docs_in_s4 (S4 confirmed flag)
  • Drop-in include _order_details.html + .js-open-orders buttons

More drop-in components

Same include-and-done philosophy (each documented in docs/claude/coordination.md and the components card of docs/screens.html):

  • Next-step archive — _next_step_history.html, ↻ archive + 🕘 history per entity
  • Teams chats — registry (teams_chats) + per-ticket refs + floating 💬 widget
  • ✉️ Issue messages — _issue_message.html + fixed special texts (issue_messages.py)
  • Comments dialog — _comment_history_dialog.html on the Retail / Spillover / ECOM boards
  • ⚠ Row validations — _row_validation_dialog.html + rule registry row_validations.py
  • Entity links (per-entity URLs) and Entity connections (many-to-many topic ↔ defect / retail / ecom / spillover, detail pages, collapsed when empty)
  • Global 🔍 search — floating widget on every page, source registry in db/search.py

Inbox + filing

Unfiled items are notes rows with entity_type='input' and entity_id='inbox'. Filing is a single SQL UPDATE:

  • UPDATE notes SET entity_type=?, entity_id=? WHERE id=?
  • All attachments follow automatically (linked to note_id)
  • Filing to Shelf additionally creates a new shelf row first, then re-parents the note to it

How to add a new module (the tracker is the reference implementation)

  1. Own storage module: a file in app/db/ (or app/db_<name>.py) with the schema, init_schema, and ALL its SQL
  2. Own Flask Blueprint app/web_<name>.py; register it (and call init_schema) in app/web.py
  3. Templates assembled from _macros.html + the style.css components
  4. Notes: one entry in web_notes.REGISTRY + {% include '_notes_section.html' %}
  5. Tests for the logic in tests/test_<name>_*.py; then update docs/claude/*.md, docs/screens.html, docs/database_schema.html, CLAUDE.md

A new import vertical (Omni) additionally follows the ECOM example: own header-map importer + own table + annotations table, wired into app/importer.py, buckets via the shared status_mappings.yaml if it reports.

Database design principles

Imported vs authored data separation

Every imported table (defects, spillover, retail) has a paired annotations table (defect_annotations, etc.). Import runs upsert the primary table; they never touch the annotations table. A user's next step or flag is never overwritten by a re-import.

Migrations via ALTER TABLE

When a new column is added to an existing table, it is added via ALTER TABLE ... ADD COLUMN in a migration block at the bottom of init_db(). These are safe to re-run (they use IF NOT EXISTS equivalents or are wrapped in try/except). Column additions are tagged migration in the schema doc.

All datetimes as TEXT (ISO 8601)

SQLite has no native datetime type. All timestamps are stored as ISO 8601 TEXT strings (YYYY-MM-DDTHH:MM:SS). Sorting and comparison work correctly with TEXT comparison on ISO strings.

Entity-type + entity-id addressing

The shared tables (notes, order_details) use a (entity_type TEXT, entity_id TEXT) pair as the address instead of typed foreign keys. This lets any new module plug into shared infrastructure without a schema change. entity_id is always the TEXT representation of the primary key.

Report outputs — PowerPoint replaced PDF

PDF is fully retired (WeasyPrint's GTK dependencies never loaded on Windows; pdf_utils.py and all PDF routes are deleted). Browser Print → Save as PDF remains the manual fallback.

Live outputs: Retail + Spillover Status Report PPTs (ppt_retail.py / ppt_spillover.py on shared ppt_utils.py primitives — direct-build, no template file) · Download-HTML snapshots on the reports and tracker boards (standalone, CSS inlined) · POST /export-reports writes dated HTML + PPTX for both reports to report_export/ for automation pickup · 📧 email reports via GMX SMTP (emailer.py, recipients/lists in the DB) · append-only Excel report log (output/retail_report_log.xlsx, Retail + ECOM sheets).

How to start the app

  • Windows (normal use): double-click run_web.bat — kills any existing process on port 8010, then starts Flask
  • Command line: python -m app.web from the project root
  • URL: http://127.0.0.1:8010
  • Database: auto-created at data/test_coordination.db on first start via database.init_db()
